# CSS Module 01 — Introduction & The Style Rule

## 1.1 What Is CSS?

**CSS — Cascading Style Sheets** — is the language used to describe the presentation of HTML documents. It controls:

- Colours, fonts, and spacing
- Layout and positioning
- Animations and transitions
- Responsive behaviour across device sizes
- Print styles

CSS was first proposed by Håkon Wium Lie in 1994. The "Cascading" in the name refers to the set of rules that determine which style wins when multiple rules could apply to the same element — a concept fundamental to understanding CSS.

---

## 1.2 The CSS Style Rule

The **style rule** is the fundamental unit of CSS. Every rule has two parts:

```
selector {
  property: value;
  property: value;
}
   │              │
   │              └── Declaration block
   └── Selector: which elements to style
```

### Full Example

```css
h1 {
  color: #1a1a2e;
  font-size: 2.5rem;
  font-weight: 700;
  line-height: 1.2;
  margin-bottom: 1rem;
}
```

- `h1` — the **selector** (targets all `<h1>` elements)
- `{ ... }` — the **declaration block**
- `color`, `font-size`, etc. — **properties**
- `#1a1a2e`, `2.5rem`, etc. — **values**
- Each property-value pair is a **declaration**
- Declarations end with a **semicolon** (`;`)
- The final semicolon before `}` is optional but recommended

## 1.3 Three Ways to Add CSS

### 1. Inline Styles

CSS written directly on an element with the `style` attribute:

```html
<h1 style="color: #1a1a2e; font-size: 2.5rem;">Welcome</h1>
<p style="margin: 1rem 0; line-height: 1.6;">Paragraph text</p>
```

**Pros:**
- Highest specificity (overrides everything except `!important`)
- Useful for dynamic styles generated by JavaScript or Thymeleaf

**Cons:**
- Mixes presentation with structure — maintenance nightmare
- Cannot use media queries, pseudo-classes, or pseudo-elements
- Cannot be cached separately by the browser
- Extremely hard to override

**When to use:** Only when generating styles dynamically from server data (e.g., a chart colour from the database). Never for general styling.

```html
<!-- Acceptable: dynamic brand colour from database -->
<div class="brand-header" 
     th:style="'background-color: ' + ${company.brandColor} + ';'">
  Company Header
</div>
```

### 2. Internal (Embedded) Styles

CSS in a `<style>` element in the `<head>`:

```html
<head>
  <style>
    body {
      font-family: 'Inter', sans-serif;
      color: #333;
    }
    h1 {
      color: #1a1a2e;
    }
  </style>
</head>
```

**Pros:**
- No extra HTTP request
- Can use all CSS features (media queries, pseudo-classes)
- Useful for page-specific critical CSS

**Cons:**
- Not shared between pages — duplicated if copied
- Not cached separately from HTML
- Mixes concerns in the HTML file

**When to use:** 
- Single-page applications where there's only one HTML file
- Injecting Critical CSS for above-the-fold content (performance optimisation)
- Email templates (email clients have poor external CSS support)

### 3. External Stylesheets (Recommended)

CSS in a separate `.css` file, linked from HTML:

```html
<head>
  <link rel="stylesheet" href="/css/main.css">
</head>
```

```css
/* /css/main.css */
body {
  font-family: 'Inter', sans-serif;
  color: #333;
}
```

**Pros:**
- Browser caches the file — loaded once, reused on all pages
- Clean separation of concerns
- Easy to maintain, version, and lint
- Multiple pages share the same styles
- Works with CSS preprocessors (Sass/Less)

**Cons:**
- Extra HTTP request (mitigated by HTTP/2 multiplexing and caching)
- Render-blocking (mitigated by serving from CDN with good cache headers)

**This is the standard approach for all enterprise applications.**

### Loading External CSS in Spring Boot + Thymeleaf

```html
<!-- Always use th:href="@{}" for context-path safety -->
<link rel="stylesheet" th:href="@{/css/normalize.css}">
<link rel="stylesheet" th:href="@{/css/main.css}">
```

Files live in `src/main/resources/static/css/` and Spring Boot serves them as-is.

## 1.4 CSS Comments

```css
/* This is a single-line comment */

/*
 * This is a multi-line comment.
 * Comments are stripped during minification.
 * Document non-obvious decisions here.
 */

/* =============================================
   Section: Typography
   ============================================= */

.btn-primary {
  background-color: var(--color-primary); /* uses CSS custom property */
}
```

Comments are removed by CSS minifiers and don't affect file size in production.

## 1.5 The Browser Default Stylesheet

Every browser has a built-in **user agent stylesheet** that applies default styles before any of your CSS. This is why:

- `<h1>` is large and bold by default
- `<ul>` has left padding and bullets
- `<a>` is blue and underlined
- `<strong>` is bold

These defaults vary slightly between browsers. **Normalize.css** or a CSS reset smooths out these differences (covered in Module 14).

### Viewing Browser Defaults in DevTools

In Chrome DevTools:
1. Inspect an element
2. Look in the **Styles** panel
3. You'll see `user agent stylesheet` entries for default styles

---

## 1.6 The CSS Processing Order

When a browser loads a page with CSS, it follows this process:

1. **Collect all stylesheets** (external, internal, inline)
2. **Parse** each rule
3. **Resolve conflicts** using the cascade (specificity, source order, `!important`)
4. **Inherit** inheritable properties from parent elements
5. **Calculate computed values** (e.g., convert `2em` to pixels based on font size)
6. **Apply** the final styles

Understanding steps 3 and 4 (cascade and inheritance) is the key to becoming proficient in CSS. They are covered in depth in Module 03.

---

## Key Takeaways

- A CSS rule = selector + declaration block (property-value pairs)
- Use external stylesheets for enterprise applications; inline for dynamic values only
- Browsers have default styles that vary — use Normalize.css to level the playing field
- CSS is processed through parsing → cascade resolution → inheritance → computed values
